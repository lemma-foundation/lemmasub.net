Verifier-grounded data subnet

Verifier-grounded training data.

Lemma v1 is a Bittensor subnet where miners produce Lean proofs accepted by a deterministic verifier, and validators publish those accepted proofs as graph-shaped public training data.

AI can guess. Verifiers can check. Lemma pays for checked data.

GitHub How it works

The mechanism

Tasks, proofs, deterministic checks, public graph rows.

Lemma turns Lean proof search into public training data: publish theorem tasks, search for accepted proofs, check them with a pinned verifier, reward verified units, publish graph-shaped corpus rows, and use that data to train stronger reasoning models.

  1. TasksValidators expose exact Lean theorem targets.
  2. ProofsMiners use any search stack they want.
  3. ChecksThe verifier decides pass or fail.
  4. DataAccepted proofs become public graph rows.

Lean first

Math is the wedge. Verified data is the product.

Lean is the first clean instance because its verifier is deterministic, mature, and backed by Mathlib. A model can write something that looks like a proof. Lean checks whether it really follows from the theorem statement and allowed imports.

Corpus

The graph-shaped corpus is the product.

Each accepted row records the Lean task, verifier, proof, identity, source, license, attribution, quality gates, dependencies, and replay information. The graph structure is foundational even while v1 rewards stay Lean-only.

{
  "schema_version": 2,
  "task_id": "lemma.sample.true_intro",
  "domain_id": "lean",
  "verifier_id": "lake-build",
  "accepted_artifact": {
    "proof": "trivial",
    "proof_identity_strength": "strong"
  },
  "dependencies": { "mathlib_imports": ["Mathlib"] },
  "graph": { "node_ids": { "task": "...", "proof": "..." } },
  "verification": { "accepted": true }
}

Model miners

Lemma data feeds model competition.

Affine-style model miners can train on Lemma's public corpora. Affine rewards model dominance; Lemma produces the verifier-grounded corpora that help reasoning models improve.

uv run lemma export-corpus \
  --domain lean \
  --format jsonl \
  --out data/lean_corpus.jsonl

Graph substrate

Every mechanism should build on the graph.

Lean is the only production launch domain. The substrate shape is already graph-native: tasks, proofs, sources, licenses, proof identities, verifier runs, solver attribution, validator attribution, imports, and future row dependencies are linked instead of treated as isolated JSON records.

  1. IdentityDuplicate families and reward eligibility.
  2. SourceLicense and provenance paths.
  3. QualityDifficulty, novelty, and usefulness gates.
  4. ReuseFuture proof dependencies and retrieval.

Validation

Validators check artifacts, not claims.

Validators enforce Lean task binding, dispatch to the pinned Lean verifier, reject duplicates, score the first accepted unique proof for each active slot, and publish replayable graph rows. Unsolved-slot value is not redistributed to current solvers.

uv run lemma worker --check
uv run lemma validate --once --no-set-weights

Benchmarks

Benchmarks measure progress. They are not the product.

The corpus trains models. Benchmarks show whether those models improve. Formal Conjectures, lean-eval, miniF2F, PutnamBench, and the IMO Grand Challenge are evaluation targets, not the v1 reward path.

Research context

AlphaProof-style ideas, open corpus mechanics.

AlphaProof-style systems showed the power of large formal curricula and verifier-grounded feedback. Lemma takes the data flywheel open: a market for verified training data, not a closed lab dataset.

  • AlphaProof and AlphaGeometry research
  • LeanNavigator state-graph generation
  • LeanConjecturer
  • LeanDojo

Lemma is independent. AlphaProof, Google DeepMind Formal Conjectures, miniF2F, PutnamBench, LeanDojo, LeanNavigator, Affine, Verus, and the IMO Grand Challenge are referenced as research context, integration context, roadmap context, or evaluation targets only. No endorsement is implied.
