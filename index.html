Protocol

Keep the reward rule simple.

Lemma publishes a theorem target. Miners produce Lean proof files out of band, submit the proof locally, and optionally serve stored proofs to validators. Validators run the pinned Lean checker. A proof either verifies or it does not.

01

Target

Everyone works against the same theorem statement and pinned Mathlib revision.

02

Proof

The miner submits a Lean file in the expected Submission namespace.

03

Winner

The first proof that Lean accepts becomes the winner for that target.

Proof file

The protocol rewards a checked file, not a story about the file.

Informal reasoning can still help humans understand a result, but the live reward path is the formal proof. Lean checks the exact theorem statement. Shortcuts like sorry or new axioms are rejected. 

import Mathlib

namespace Submission

theorem nat_two_plus_two_eq_four : (2 : Nat) + 2 = 4 := by
  rfl

end Submission

Miners

Generate proofs however you want.

Use a proving model, a local search loop, a notebook, or manual work. The protocol only needs the finished Lean file.

Validators

Check the proof, then publish the winner.

Validators poll proof-serving miners, run the pinned verifier, reject invalid files, and assign the winner-take-all reward signal.

Miner Guide

Direction

Build proof capacity first.

The useful thing to grow is the ability to turn mathematical targets into Lean-verified proofs. The public protocol stays narrow so miners can focus on that work.

01

Manual submissions

Proof generation stays out of band. The CLI stores and serves verified proofs.

02

Objective checking

Lean decides whether the proof is valid. There is no prose score.

03

Harder targets

As tooling improves, the target supply can move toward more valuable theorem work.
