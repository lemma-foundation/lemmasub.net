Manual bounties

Bounties are not live yet.

Lemma is preparing manual reviewed rewards for harder Lean proof work. These will be separate from cadence miner rounds: no timer, no automatic validator scoring, and no active payout or claim intake yet.

The intended process is aligned with the normal Formal Conjectures and Lean community workflow: publish a pinned target, submit proof work through GitHub review, verify the exact artifact, then make a reviewed payout decision.

Note: Lemma is not affiliated with or partnered with Google DeepMind.

Current status

Informational only.

There are no active Lemma bounties, no live claim form, and no public bounty treasury payout path. This page exists so solvers know what will be required before any high-value bounty opens.

Future process

Manual review before any payout.

Cadence rounds are routine: miners receive theorem variants, validators check proof files, and passing proofs become reward-eligible. Bounties are different. They may involve longer proofs, helper lemmas, supporting files, upstream review, and checks that the formal statement matches the intended target.

Publish a locked target

Lemma will publish a bounty record before work starts, including the source URL, pinned repository commit or tag, Lean and Mathlib versions, target statement hash, reward, creation timestamp, and whether formal_proof already existed.

Use public review

Solvers should work through a GitHub PR, issue, or external proof repository. For Formal Conjectures targets, that means following the project contribution process and producing reproducible lake build evidence.

Claim with structure

A TAO address comment in Lean code can be useful metadata, but it is not enough. The real claim must live in a structured GitHub issue, PR, proof repository, or Lemma bounty registry record.

Claim record

The payout address must be tied to the claimant.

Future claims should include GitHub identity, proof provenance, TAO address, and preferably a wallet-signed message. This keeps someone from copying a proof and redirecting the reward by editing a comment.

Wallet signature: optional for small bounties, required for high-value bounties

Review gates

High-value bounties need reproducible public evidence.

Before payout, Lemma should verify the proof in the pinned environment, match the exact target hash, reject shortcuts such as sorry, admit, new axioms, unsafe hooks, or unauthorized imports, check that the proof was not public before the bounty was created, and confirm claimant identity.

Future claim statuses are: open, submitted, lemma_verified, upstream_review, accepted_formal_proof, paid, rejected, and disputed.

Normal proof bounties should exclude targets that already had a Formal Conjectures formal_proof at bounty creation. Those targets can only be offered as explicit proof-porting or simplification work.
